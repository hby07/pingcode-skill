# 头脑风暴与产品设计工作流

> 对应 Phase 1: 产品设计。输出 PRD 文档。

---

## 概述

本工作流引导用户从模糊的想法出发，通过头脑风暴逐步收敛为结构化的 PRD 文档，最终推送到 PingCode Wiki 供团队审核。

**核心原则**：发散 → 收敛 → 文档化 → 审核

---

## 前置条件

| 条件 | 说明 |
|------|------|
| PingCode 连通 | 环境变量已配置，API 可调用 |
| Wiki 空间已创建 | 或准备在流程中创建 |

---

## 工作流步骤

### Step 1: 需求采集

**目标**：理解用户想做什么

```
❓ 请描述你想做的产品/功能：

（可以自由输入，包括：
- 你想解决什么问题？
- 目标用户是谁？
- 你设想的核心功能是什么？
- 有没有参考产品？）
```

**Agent 行为**：
- 认真倾听，不打断
- 通过追问澄清模糊点
- 记录关键信息

**追问示例**：
- "你说的 XX 具体是指什么？"
- "目标用户是个人还是企业？"
- "有没有你觉得做得好的类似产品？"

---

### Step 2: 头脑风暴

**目标**：发散想法，探索可能性

**Agent 行为**：
1. 基于用户输入，提出 3-5 个方向/方案
2. 每个方向列出核心功能点
3. 指出各方向的优劣

**输出格式**：

```
🧠 基于你的描述，我想到几个方向：

### 方向 A：{名称}
- 核心功能：...
- 优势：...
- 劣势：...

### 方向 B：{名称}
- 核心功能：...
- 优势：...
- 劣势：...

### 方向 C：{名称}
- 核心功能：...
- 优势：...
- 劣势：...

❓ 你倾向哪个方向？或者想组合几个方向的特点？

A. 方向 A
B. 方向 B
C. 方向 C
D. 组合（请说明）
E. 都不是，我有其他想法
```

---

### Step 3: 范围界定

**目标**：确定 MVP 范围

```
❓ 确认 MVP 范围：

核心功能（必须有）：
1. {功能1}
2. {功能2}
3. {功能3}

可以有（第二期）：
1. {功能4}
2. {功能5}

不做（明确排除）：
1. {排除项1} - 原因：...
2. {排除项2} - 原因：...

❓ 范围是否合适？

[确认范围] [调整范围] [重新讨论]
```

---

### Step 4: 架构草图

**目标**：初步确定系统架构

**Agent 行为**：
1. 画出系统架构图（文本格式）
2. 说明各组件职责
3. 标注技术选型建议

**输出格式**：

```
📐 初步架构：

┌─────────────┐     ┌─────────────┐
│  组件 A      │────→│  组件 B      │
│  (职责)      │     │  (职责)      │
└─────────────┘     └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  组件 C      │
                    │  (职责)      │
                    └─────────────┘

技术选型建议：
- 后端：{建议 + 理由}
- 前端：{建议 + 理由}
- 数据库：{建议 + 理由}
```

---

### Step 5: PRD 编写

**目标**：将讨论结果整理为 PRD 文档

**Agent 行为**：
1. 按 PRD 模板结构编写文档
2. 包含所有讨论确定的内容
3. 标注开放问题和待决策项

**PRD 结构**：
```markdown
# {产品名} — 产品需求文档 (PRD)

## 1. 产品概述
## 2. 目标用户与使用场景
## 3. 系统架构
## 4. 功能需求
## 5. 非功能需求
## 6. 里程碑与排期
## 7. 开放问题
```

---

### Step 6: 本地审核

**目标**：用户在本地审核 PRD

```
📝 PRD 已写入本地文件：{文件路径}

请你审核以下内容：
1. 产品定位是否准确？
2. 功能需求是否完整？
3. 技术方案是否合理？
4. 有没有遗漏的场景？

审核完成后告诉我：
- "没问题" → 进入 Step 7
- "需要修改" → 说明修改点，我来更新
```

---

### Step 7: 推送 Wiki

**目标**：将 PRD 推送到 PingCode Wiki

**前置确认**：

```
⚠️ 即将执行以下操作：

1. 创建/使用 Wiki 空间：{空间名}
2. 创建页面：{页面名}
3. 内容：PRD 文档（{字数} 字）

❓ 确认推送到 PingCode Wiki？

[确认推送] [先不推送] [取消]
```

**用户确认后执行**：
1. 创建或确认 Wiki 空间
2. 添加用户为成员
3. 创建 Wiki 页面
4. 写入 PRD 内容

---

### Step 8: 完成与引导

```
✅ Phase 1（产品设计）完成！

📊 交付物：
- PRD 文档：PingCode Wiki → {空间名} → {页面名}
- 本地副本：{文件路径}

📝 下一步建议：
进入 Phase 2（技术设计），基于 PRD 编写技术方案。

❓ 是否继续进入技术设计？

[继续技术设计] [稍后再说] [查看 PRD]
```

---

## 注意事项

- 头脑风暴阶段鼓励发散，不要过早否定想法
- 范围界定阶段要勇于说"不做"，MVP 越小越好
- PRD 不需要写得完美，够用就行，后续可以迭代
- 开放问题要如实记录，不要假装已经想清楚了
- 推送 Wiki 前必须获得用户确认
