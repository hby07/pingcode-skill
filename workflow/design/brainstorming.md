# 产品设计协作工作流

> Agent 作为产品设计助手，引导用户完成从想法到 PRD 的全过程。

---

## 触发条件

用户说以下任一：
- "帮我设计产品"
- "头脑风暴一下"
- "我想做一个新产品"
- "帮我写 PRD"
- "设计一个功能"

---

## 工作流步骤

```
[⬜] Step 1: 启动设计会话
[⬜] Step 2: 需求收集
[⬜] Step 3: 用户画像
[⬜] Step 4: 功能规划
[⬜] Step 5: 优先级排序
[⬜] Step 6: 生成 PRD
[⬜] Step 7: 写入 PingCode
```

---

## Step 1: 启动设计会话

### 交互
```
🎨 产品设计协作模式

欢迎！我将作为你的产品设计助手，协助你完成产品设计。

在开始之前，请告诉我：

1. 你要设计什么产品？（简要描述）
   输入：___

2. 目标用户是谁？
   A. 企业用户
   B. 个人用户
   C. 开发者
   D. 其他：___

3. 核心要解决的问题是什么？
   输入：___

确认后，我将引导你完成：用户画像 → 需求收集 → 功能规划 → PRD
```

---

## Step 2: 需求收集

### 方法：结构化提问

### 交互
```
💡 让我们来深入了解一下你的产品。

Q1: 这个产品的主要使用场景是什么？
    A. 工作办公
    B. 生活娱乐
    C. 学习教育
    D. 其他：___

Q2: 用户会通过什么设备使用？
    A. 电脑（Web/Desktop）
    B. 手机（iOS/Android）
    C. 平板
    D. 多端支持

Q3: 有没有类似的产品可以参考？
    输入：___

Q4: 这个产品的核心差异化是什么？
    输入：___

（你可以自由回答，或告诉我你的想法）
```

### 需求收集维度

| 维度 | 问题 |
|------|------|
| 用户 | 谁会使用？使用频率？ |
| 场景 | 什么情况下使用？ |
| 痛点 | 现有方案有什么问题？ |
| 价值 | 能给用户带来什么？ |
| 竞品 | 有哪些类似产品？ |
| 差异 | 我们有什么不同？ |

## Step 4: 功能规划

### 交互
```
📋 功能规划

根据用户画像，我建议以下功能模块：

核心功能（MVP）：
├── F1: {功能1} - {价值}
├── F2: {功能2} - {价值}
└── F3: {功能3} - {价值}

扩展功能（V2）：
├── F4: {功能4} - {价值}
└── F5: {功能5} - {价值}

未来规划（V3）：
└── F6: {功能6} - {价值}

📊 功能优先级：
1. {功能} - {理由}
2. {功能} - {理由}
3. {功能} - {理由}

❓ 请问：
1. 功能规划是否合理？
2. 是否需要增减功能？
3. 优先级是否需要调整？

[确认规划] [调整功能] [重新规划]
```

### 功能规划方法

#### MoSCoW 方法
- **Must have**：核心功能，没有就不能用
- **Should have**：重要功能，但不是核心
- **Could have**：锦上添花，有更好
- **Won't have**：本期不做

#### 用户故事地图
```
用户旅程：  登录 → 浏览 → 操作 → 查看结果
           ─────────────────────────────────
Must have:  F1    F2     F3     F4
Should have:      F5     F6     F7
Could have:       F8            F9
```

## Step 5: 优先级排序

### 交互
```
📊 功能优先级排序

我根据以下维度进行了排序：

| 功能 | 用户价值 | 开发成本 | 优先级 |
|------|----------|----------|--------|
| F1 | 高 | 低 | P0 |
| F2 | 高 | 中 | P0 |
| F3 | 中 | 低 | P1 |
| F4 | 中 | 高 | P2 |
| F5 | 低 | 低 | P2 |

排序依据：
- 用户价值：对用户的重要性
- 开发成本：实现难度和工时
- 依赖关系：是否依赖其他功能

❓ 请问：
1. 优先级是否合理？
2. 是否需要调整？

[确认优先级] [调整排序] [重新评估]
```

## Step 7: 写入 PingCode Wiki

### 操作
- 将 PRD 写入 PingCode Wiki（文档空间）
- 这是**文档**，不是正式需求工作项

### 交互
```
✅ PRD 已写入 PingCode Wiki！

📊 执行结果：
- ✅ 文档标题：{产品名称} - 产品需求文档
- ✅ 存储位置：PingCode Wiki
- ✅ 文档状态：草稿

⚠️ 重要提醒：
当前写入的是 Wiki 文档（供团队阅读和审核），不是正式的需求工作项。

下一步流程：
1. 请团队成员审核 Wiki 文档
2. 收集反馈意见
3. 审核通过后，再创建正式的需求工作项

❓ 请问：
1. 是否需要通知团队审核？
2. 是否继续完善文档？
3. 审核通过后，我可以帮你创建正式的需求工作项

[通知团队] [继续完善] [等待审核] [结束]
```

## Step 8: 审核通过后创建需求（可选）

### 前提
- Wiki 文档已审核通过
- 用户确认要创建正式需求

```
📋 Wiki 文档审核已通过，准备创建正式需求。

我将从 Wiki 文档中提取以下信息创建需求工作项：

| 字段 | 值 |
|------|-----|

⚠️ 以上操作将写入 PingCode：
- 创建 {N} 个需求工作项
- 关联 Wiki 文档

❓ 确认创建？

[确认创建] [修改信息] [取消]
```

## 完整流程总结

```
Step 1: 启动设计会话
Step 2: 需求收集
Step 3: 用户画像
Step 4: 功能规划
Step 5: 优先级排序
Step 6: 生成 PRD
Step 7: 写入 PingCode Wiki（文档）  ← 这里
    ↓
  [人工审核]
    ↓
Step 8: 审核通过后创建需求工作项    ← 这里
```

**关键原则：先文档，后需求。文档审核通过后，才创建正式需求工作项。**

## 头脑风暴技巧

### 引导性问题

#### 发散阶段
- "还有呢？"
- "如果不受限制，你会怎么做？"
- "用户最想要什么？"
- "竞争对手做了什么？"

#### 收敛阶段
- "这真的是用户需要的吗？"
- "这个功能的核心价值是什么？"
- "如果只能做三个功能，你选哪三个？"

### 创意方法

#### SCAMPER
- **S**ubstitute（替代）
- **C**ombine（合并）
- **A**dapt（调整）
- **M**odify（修改）
- **P**ut to other uses（其他用途）
- **E**liminate（消除）
- **R**everse（反转）

#### 六顶思考帽
- ⚪ 白帽：事实和数据
- 🔴 红帽：直觉和情感
- ⚫ 黑帽：风险和问题
- 🟡 黄帽：价值和好处
- 🟢 绿帽：创意和替代
- 🔵 蓝帽：流程和控制
