# 技术设计工作流

> 基于已审核的 PRD，进行技术方案设计并输出技术文档。

---

## 触发条件

用户说以下任一：
- "写技术设计"
- "做技术方案"
- "设计技术架构"
- "技术评审"

---

## ⚠️ 前置条件

- ✅ PRD 已完成并审核通过
- ✅ 需求工作项已创建（可选）

---

## 工作流步骤

```
[⬜] Step 1: 读取 PRD
[⬜] Step 2: 分析技术需求
[⬜] Step 3: 架构设计
[⬜] Step 4: 接口设计
[⬜] Step 5: 数据库设计
[⬜] Step 6: 生成技术文档
[⬜] Step 7: 写入 PingCode Wiki
[⬜] Step 8: 引导进入下一阶段
```

---

## Step 1: 读取 PRD

### 操作
- 从 PingCode Wiki 读取 PRD 文档
- 提取功能需求和非功能需求

### 交互
```
📋 正在读取 PRD 文档...

找到以下 PRD：
| # | 文档标题 | 创建时间 | 状态 |
|---|----------|----------|------|
| 1 | {PRD标题} | {时间} | 已审核 |

📊 PRD 概要：
- 产品名称：{名称}
- 功能数量：{N} 个
- 核心需求：{摘要}

❓ 请问：
1. 要基于这个 PRD 进行技术设计吗？
2. 是否需要补充其他需求文档？

[确认开始] [选择其他PRD] [取消]
```

---

```
基于 PRD，我识别出以下技术需求：

2. {功能2} - {技术实现要点}

非功能需求：
- 性能：{要求}
- 安全：{要求}
- 可用性：{要求}

技术约束：
- {约束1}
- {约束2}

❓ 请问：
1. 技术需求分析是否准确？
2. 是否需要补充其他约束？

[确认分析] [调整分析] [取消]
```

## Step 3: 架构设计

### 交互
```
🏗️ 架构设计：

我建议采用以下架构：

系统架构：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端      │ ──→ │   后端      │ ──→ │   数据库    │
│  (React)    │     │  (Node.js)  │     │ (PostgreSQL)│
└─────────────┘     └─────────────┘     └─────────────┘

模块划分：
| 模块 | 职责 | 技术栈 |
|------|------|--------|
| {模块1} | {职责} | {技术} |
| {模块2} | {职责} | {技术} |

- {方案描述}

❓ 请问：
1. 架构方案是否合理？
2. 是否需要调整技术选型？

[确认架构] [调整方案] [重新设计]
```

## Step 4: 接口设计

### 交互
```
🔌 接口设计：

基于功能需求，设计以下接口：

| 接口 | 方法 | 路径 | 描述 |
|------|------|------|------|
| 登录 | POST | /api/auth/login | 用户登录 |
| 获取用户 | GET | /api/users/:id | 获取用户信息 |
| ... | ... | ... | ... |

接口详情示例：

POST /api/auth/login
请求：
{
  "username": "string",
  "password": "string"
}
响应：
{
  "code": 0,
  "data": {
    "token": "string",
    "user": {...}
  }
}

❓ 请问：
1. 接口设计是否完整？
2. 是否需要调整接口规范？

[确认接口] [调整接口] [取消]
```

## Step 7: 写入 PingCode Wiki

### 操作
- 将技术文档写入 PingCode Wiki
- 关联 PRD 文档

### 交互
```
⚠️ 确认操作：

我将执行以下操作：
1. 创建技术设计文档到 PingCode Wiki
2. 关联 PRD 文档：{PRD链接}

❓ 确认写入？

[确认写入] [修改信息] [取消]
```

### 交互（写入完成）
```
✅ 技术设计文档已写入 PingCode Wiki！

📊 执行结果：
- ✅ 文档标题：{产品名称} - 技术设计文档
- ✅ 存储位置：PingCode Wiki
- ✅ 关联文档：{PRD链接}
- ✅ 文档状态：草稿

⚠️ 重要提醒：
当前写入的是 Wiki 文档（供团队审核），请组织技术评审。

📝 后续步骤：
1. 组织技术评审会议
2. 收集团队反馈
3. 审核通过后，进入任务拆解阶段

❓ 请问：
1. 是否需要通知团队评审？
2. 是否继续完善文档？

```

## 技术设计规范

### 架构设计要点
- 高内聚、低耦合
- 可扩展性
- 可维护性
- 安全性

### 接口设计规范
- RESTful 风格
- 统一响应格式
- 错误码规范
- 版本管理

### 数据库设计规范
- 第三范式
- 适当冗余
- 索引优化
- 数据安全
