# 技术设计工作流

> 对应 Phase 2: 技术设计。输出技术设计文档。

---

## 概述

本工作流基于已审核的 PRD，引导用户完成技术方案设计，输出技术设计文档并推送到 PingCode Wiki。

**核心原则**：PRD 驱动 → 方案讨论 → 文档化 → 审核

---

## 前置条件

| 条件 | 说明 |
|------|------|
| PRD 已完成 | Phase 1 的 PRD 文档已审核通过 |
| PingCode Wiki 空间 | 已有产品文档空间（复用 Phase 1 的空间） |

---

## 工作流步骤

### Step 1: 读取 PRD

**目标**：理解产品需求，为技术设计做准备

**Agent 行为**：
1. 读取 PRD 文档（本地文件或 PingCode Wiki）
2. 提取关键信息：功能需求、非功能需求、系统架构、开放问题
3. 梳理技术挑战点

**输出格式**：

```
📖 已读取 PRD，关键信息摘要：

产品：{产品名}
核心功能：{N} 个模块
技术栈：{已有建议或待定}

📌 技术挑战点：
1. {挑战1} — {为什么难}
2. {挑战2} — {为什么难}

❓ 在开始技术设计前，有几个问题需要确认：

1. {问题1}
2. {问题2}
```

---

### Step 2: 技术选型讨论

**目标**：确定技术栈和关键框架

**Agent 行为**：
1. 基于 PRD 中的技术建议，给出选型方案
2. 列出各方案的优劣
3. 让用户决策

**输出格式**：

```
🔧 技术选型：

❓ 后端框架：
A. {框架1} — {特点}
B. {框架2} — {特点}
C. {框架3} — {特点}

❓ 数据存储：
A. {方案1} — {特点}
B. {方案2} — {特点}

❓ 前端/客户端：
A. {方案1} — {特点}
B. {方案2} — {特点}

请输入每项的选择（如：后端选 A，存储选 B）：___
```

---

### Step 3: 架构设计

**目标**：确定系统整体架构

**Agent 行为**：
1. 画出详细的系统架构图
2. 说明各组件职责和交互方式
3. 标注数据流向

**输出格式**：

```
📐 系统架构：

┌──────────────────────────────────────────┐
│              {组件名}                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 模块 A   │  │ 模块 B   │  │ 模块 C   │  │
│  └────┬────┘  └────┬────┘  └────┬────┘  │
│       └────────────┼────────────┘        │
│              ┌─────┴─────┐               │
│              │  共享层    │               │
│              └─────┬─────┘               │
└────────────────────┼─────────────────────┘
                     │ {协议}
                     ▼
┌──────────────────────────────────────────┐
│              {组件名}                     │
└──────────────────────────────────────────┘

组件职责：
- {组件1}：{职责描述}
- {组件2}：{职责描述}
- {组件3}：{职责描述}
```

---

### Step 4: 接口设计

**目标**：定义系统间/模块间的接口

**Agent 行为**：
1. 列出所有需要的接口
2. 定义请求/响应格式
3. 说明错误处理方式

**输出格式**：

```
📡 接口设计：

### {接口组名}

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/xxx | {说明} |
| GET  | /api/xxx | {说明} |

### {具体接口名}

请求：
{
  "field": "type — 说明"
}

响应：
{
  "field": "type — 说明"
}

错误码：
| 状态码 | 说明 |
|--------|------|
| 400 | {说明} |
```

---

### Step 5: 数据模型设计

**目标**：定义数据结构和存储方案

**Agent 行为**：
1. 设计核心数据模型
2. 说明存储方案
3. 标注索引和查询模式

**输出格式**：

```
💾 数据模型：

### {模型名}

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 自动 | 唯一标识 |
| name | string | 是 | {说明} |
| created_at | timestamp | 自动 | 创建时间 |

存储方案：{SQLite / JSON / Redis / ...}
索引：{按 XX 查询}
```

---

### Step 6: 文档编写

**目标**：将技术方案整理为完整的技术设计文档

**文档结构**：
```markdown
# {产品名} — 技术设计文档

## 1. 概述
## 2. 技术选型
## 3. 系统架构
## 4. 接口设计
## 5. 数据模型
## 6. 部署方案
## 7. 安全设计
## 8. 开放问题
```

---

### Step 7: 本地审核

```
📝 技术设计文档已写入本地：{文件路径}

请审核：
1. 架构设计是否合理？
2. 接口定义是否完整？
3. 数据模型是否满足需求？
4. 有没有遗漏的技术点？

[审核通过] [需要修改] [取消]
```

---

### Step 8: 推送 Wiki

```
⚠️ 即将推送到 PingCode Wiki：

- 空间：{空间名}
- 页面：{页面名}
- 内容：技术设计文档（{字数} 字）

❓ 确认推送？

[确认推送] [修改后推送] [取消]
```

---

### Step 9: 完成与引导

```
✅ Phase 2（技术设计）完成！

📊 交付物：
- 技术设计文档：PingCode Wiki → {页面名}
- 本地副本：{文件路径}

📝 下一步建议：
进入 Phase 3（任务拆解），基于 PRD + 技术设计拆解开发任务。

❓ 是否继续进入任务拆解？

[继续拆解] [稍后再说]
```

---

## 注意事项

- 技术选型要考虑团队熟悉度，不追求最新技术
- 架构设计先粗后细，不要一开始就陷入细节
- 接口设计要考虑向前兼容
- 数据模型要预留扩展字段
- 开放问题要如实记录，技术设计不是一步到位的
