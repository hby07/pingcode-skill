# Sprint 计划工作流

> 将产品需求拆解为可执行的 Sprint 计划，遵循 Scrum 规范。

---

## 触发条件

用户说以下任一：
- "帮我做 Sprint 计划"
- "创建迭代计划"
- "规划下一个 Sprint"
- "把需求分配到迭代"

---

## 工作流步骤

```
[⬜] Step 1: 读取需求
[⬜] Step 2: 分析与排序
[⬜] Step 3: 选择方法论
[⬜] Step 4: 拆解任务
[⬜] Step 5: 估算工时
[⬜] Step 6: 分配 Sprint
[⬜] Step 7: 预览确认
[⬜] Step 8: 写入 PingCode
```

---

## Step 1: 读取需求

### 操作
- 从 PingCode 读取项目需求列表
- 按优先级排序
- 统计需求数量和预估工时

### 交互
```
📋 我读取到了以下需求：

| # | 需求名称 | 优先级 | 预估工时 | 状态 |
|---|----------|--------|----------|------|
| 1 | {需求1} | P0 | 3天 | 待开发 |
| 2 | {需求2} | P1 | 5天 | 待开发 |
| 3 | {需求3} | P2 | 2天 | 待开发 |

📊 共 {N} 个需求，预估 {M} 人天

❓ 请问：
1. 是否需要调整优先级？
2. 是否有需求需要拆解或合并？
3. 确认后我将进入下一步「分析与排序」

[确认继续] [调整需求] [取消]
```

### 确认点
- 用户确认需求列表正确
- 用户确认优先级合理
- 用户确认无需拆分/合并

---

## Step 2: 分析与排序

### 操作
- 分析需求间的依赖关系
- 识别关键路径
- 按业务价值和风险排序

### 交互
```
🔍 需求分析结果：

依赖关系：
- 需求2 依赖 需求1（登录后才能导出）
- 需求3 无依赖

风险评估：
- 需求2：中风险（涉及第三方服务）
- 需求3：低风险

建议顺序：
1. 需求1（无依赖，高价值）
2. 需求3（无依赖，可并行）
3. 需求2（有依赖，中风险）

❓ 请问：
1. 依赖关系是否正确？
2. 风险评估是否合理？
3. 排序是否需要调整？

[确认分析] [调整分析] [取消]
```

---

## Step 3: 选择方法论

### 交互
```
🎯 请选择开发方法论：

A. Scrum（敏捷）
   - 适合：需求变化快、团队协作紧密
   - 周期：2-4 周 Sprint
   - 产出：Sprint Backlog, 每日站会, Sprint Review

B. Kanban（看板）
   - 适合：持续交付、运维场景
   - 周期：无固定周期，按需流动
   - 产出：看板视图, WIP 限制, 流程图

C. Waterfall（瀑布）
   - 适合：需求明确、文档要求高
   - 周期：阶段式推进
   - 产出：需求文档→设计文档→开发→测试→上线

请输入选项（A/B/C）：___
```

---

## Step 4: 拆解任务

### 操作（以 Scrum 为例）
- 将需求拆解为 User Story
- 将 Story 拆解为 Task
- 识别任务间依赖

### 交互
```
📝 根据 Scrum 方法论，我将需求拆解如下：

Epic: {需求名称}
├── Story 1: {故事描述} (3h)
│   ├── Task 1.1: {任务描述} (1h)
│   ├── Task 1.2: {任务描述} (1h)
│   └── Task 1.3: {任务描述} (1h)
├── Story 2: {故事描述} (2h)
│   ├── Task 2.1: {任务描述} (1h)
│   └── Task 2.2: {任务描述} (1h)
└── Story 3: {故事描述} (1h)
    └── Task 3.1: {任务描述} (1h)

📊 共 {N} 个 Story，{M} 个 Task，预估 {X}h

❓ 请问：
1. 任务拆解粒度是否合适？
2. 工时估算是否合理？
3. 是否需要调整依赖关系？

[确认拆解] [调整任务] [重新拆解]
```

### 交互
```
⏱️ 工时估算确认：

| 任务 | 估算工时 | 负责人 | 备注 |
|------|----------|--------|------|
| Task 1.1 | 1h | 待分配 | |
| Task 1.2 | 1h | 待分配 | |
| Task 1.3 | 1h | 待分配 | 需联调 |
| ... | ... | ... | ... |

📊 总计：{N}h

❓ 请问：
1. 工时估算是否合理？
2. 负责人如何分配？
   A. 自动分配（根据团队成员负载）
   B. 手动指定
   C. 先不分配

[确认估算] [调整工时] [取消]
```

## Step 6: 分配 Sprint

### 交互
```
📅 Sprint 分配建议：

Sprint 1: {开始日期} ~ {结束日期} ({时长})
├── Story 1: {故事描述} ({工时})
├── Story 2: {故事描述} ({工时})
└── 总计: {工时}

Sprint 2: {开始日期} ~ {结束日期} ({时长})
├── Story 3: {故事描述} ({工时})
├── Story 4: {故事描述} ({工时})
└── 总计: {工时}

📊 分配依据：
- Sprint 1 容量：{N}h（根据团队速率）
- Sprint 2 容量：{N}h

❓ 请问：
1. Sprint 划分是否合理？
2. 是否需要调整时间范围？
3. 是否需要调整分配？

[确认分配] [调整分配] [重新分配]
```

## Scrum 规范参考

### Sprint 周期
- 推荐：2 周
- 可选：1 周 / 3 周 / 4 周

### 任务粒度
- Story：1-3 天
- Task：0.5-2 天
- 如超过，需进一步拆分

### 估算单位
- Story Point（推荐）：1, 2, 3, 5, 8, 13, 21
- 工时（小时）：适用于小团队

### 容量规划
- Sprint 容量 = 团队人数 × 工作日 × 8h × 0.7（预留会议、沟通时间）
- 示例：5人 × 10天 × 8h × 0.7 = 280h
