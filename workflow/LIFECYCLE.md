# 产品开发生命周期

> 所有工作流必须遵循此生命周期，不得跳过阶段。

---

## 阶段总览

```
Phase 1: 产品设计    ──→  Phase 2: 技术设计    ──→  Phase 3: 任务拆解
       (PRD)                  (技术方案)                (Task Breakdown)
         │                       │                        │
         ▼                       ▼                        ▼
   Wiki 文档审核            Wiki 文档审核            PingCode 工作项
         │                       │                        │
         ▼                       ▼                        ▼
Phase 4: 开发执行    ──→  Phase 5: 测试验证    ──→  Phase 6: 发布上线
       (Sprint)                (QA)                    (Release)
         │                       │                        │
         ▼                       ▼                        ▼
   PingCode 任务状态        缺陷修复循环            版本标记 + 通知
```

---

## 各阶段定义

### Phase 1: 产品设计

| 项目 | 内容 |
|------|------|
| **目标** | 明确产品做什么、为谁做、解决什么问题 |
| **输入** | 用户想法、业务需求、市场调研 |
| **输出** | PRD（产品需求文档） |
| **存放位置** | PingCode Wiki → 产品文档空间 |
| **工作流** | `design/brainstorming.md` |
| **完成标准** | PRD 文档已写入 Wiki，用户审核通过 |
| **后续动作** | 引导用户进入 Phase 2 |

**关键活动**：
1. 头脑风暴，发散想法
2. 整理核心功能、用户场景、系统架构
3. 编写 PRD 文档
4. 推送至 PingCode Wiki
5. 用户审核确认

---

### Phase 2: 技术设计

| 项目 | 内容 |
|------|------|
| **目标** | 确定技术方案、架构设计、接口定义 |
| **输入** | 已审核的 PRD |
| **输出** | 技术设计文档（架构、接口、数据模型、部署方案） |
| **存放位置** | PingCode Wiki → 产品文档空间（与 PRD 同空间） |
| **工作流** | `design/technical_design.md` |
| **完成标准** | 技术设计文档已写入 Wiki，用户审核通过 |
| **后续动作** | 引导用户进入 Phase 3 |

**关键活动**：
1. 技术选型（语言、框架、数据库）
2. 系统架构设计
3. 接口详细定义（API 文档）
4. 数据模型设计
5. 部署方案
6. 编写技术设计文档
7. 推送至 PingCode Wiki
8. 用户审核确认

---

### Phase 3: 任务拆解

| 项目 | 内容 |
|------|------|
| **目标** | 将需求拆解为可执行的开发任务，估算工时 |
| **输入** | 已审核的 PRD + 技术设计文档 |
| **输出** | PingCode 工作项（需求 → 子任务） |
| **存放位置** | PingCode 产品/项目 → 工作项 |
| **完成标准** | 所有任务已创建，工时已估算，依赖关系已标注 |
| **前置条件** | Phase 1 + Phase 2 的文档已审核通过 |
| **后续动作** | 引导用户进入 Phase 4（Sprint 计划或看板排期） |

**关键活动**：
1. 读取 PRD 和技术设计文档
2. 按模块拆解为开发任务
3. 估算每个任务的工时
4. 标注任务间的依赖关系
5. 确定优先级
6. 批量创建 PingCode 工作项
7. 用户确认任务列表

---

### Phase 4: 开发执行

| 项目 | 内容 |
|------|------|
| **目标** | 按计划完成开发任务 |
| **输入** | 已拆解的任务列表 |
| **输出** | 可运行的代码 |
| **工作流** | `scrum/sprint_planning.md`（Scrum）或 `kanban/board_setup.md`（Kanban） |
| **完成标准** | 当前 Sprint/迭代的任务全部完成 |
| **后续动作** | 引导用户进入 Phase 5 |

**关键活动**：
1. 选择开发方法论（Scrum / Kanban）
2. 创建迭代/规划看板
3. 每日开发 + 状态更新
4. 代码审查
5. 持续集成

---

### Phase 5: 测试验证

| 项目 | 内容 |
|------|------|
| **目标** | 验证功能正确性，修复缺陷 |
| **输入** | 开发完成的功能 |
| **输出** | 通过测试的稳定版本 |
| **工作流** | `scrum/bug_triage.md` |
| **完成标准** | 所有 P0/P1 缺陷已修复，测试通过 |
| **后续动作** | 引导用户进入 Phase 6 |

**关键活动**：
1. 功能测试
2. 缺陷记录与分诊
3. 缺陷修复
4. 回归测试
5. 验收确认

---

### Phase 6: 发布上线

| 项目 | 内容 |
|------|------|
| **目标** | 将稳定版本交付给用户 |
| **输入** | 通过测试的版本 |
| **输出** | 已发布的版本 |
| **工作流** | `devops/release_management.md` |
| **完成标准** | 版本已发布，用户已通知 |
| **后续动作** | 进入下一产品迭代（回到 Phase 1） |

**关键活动**：
1. 发布准备（版本号、变更日志）
2. 部署执行
3. 发布后验证
4. 通知相关方
5. 复盘总结

---

## 阶段流转规则

| 规则 | 说明 |
|------|------|
| **不可跳过** | 每个阶段必须完成后才能进入下一阶段 |
| **可回退** | 如后续阶段发现问题，可回退到前序阶段修改 |
| **文档先行** | Phase 1-3 的输出必须是文档，审核通过后才创建工作项 |
| **用户确认** | 每个阶段完成时，必须获得用户确认才能流转 |
| **灵活适配** | 小项目可简化（如跳过 Phase 5 的正式测试），但需用户明确确认 |

---

## 阶段与 PingCode 的对应关系

| 阶段 | PingCode 模块 | 操作 |
|------|--------------|------|
| Phase 1-2 | Wiki（知识库） | 创建/更新文档页面 |
| Phase 3 | Ship（产品）/ PJM（项目） | 创建需求/任务工作项 |
| Phase 4 | Ship / PJM | 更新工作项状态、评论、工时 |
| Phase 5 | Ship / PJM | 创建缺陷、更新状态 |
| Phase 6 | Release（发布） | 创建版本、标记发布 |

| 用户意图 | 当前阶段 | 下一步 |
|----------|----------|--------|

| "上线了，开始新功能" | Phase 6 → 1 | 回到产品设计 |
