# 产品开发生命周期

> 完整的产品开发流程，从想法到发布。Agent 必须遵循此流程，不得跳过阶段。

---

## ⚠️ 核心原则

**文档先行，工作项在后。每个阶段都必须有对应的文档，审核通过后才能进入下一阶段。**

---

## 阶段间的衔接规则

### 规则 1：每个阶段完成后，引导用户进入下一阶段

```
Phase 1 完成后：
"PRD 已创建并审核通过。下一步是技术设计，要继续吗？"

Phase 2 完成后：
"技术设计文档已创建并审核通过。下一步是任务拆解和排期，要继续吗？"

Phase 3 完成后：
"Sprint 计划已创建。下一步是开始开发，要继续吗？"
```

### 规则 2：每个阶段必须有文档产出

| 阶段 | 文档产出 | 存储位置 |
|------|----------|----------|
| 产品设计 | PRD | PingCode Wiki |
| 技术设计 | 技术设计文档 | PingCode Wiki |
| 任务拆解 | Sprint 计划 | PingCode 工作项 |
| 开发执行 | 代码 | Git 仓库 |
| 测试验证 | 测试报告 | PingCode Wiki |
| 发布上线 | Release Notes | PingCode Wiki |

### 规则 3：文档审核通过后才能创建工作项

- PRD 审核通过 → 创建需求工作项
- 技术设计审核通过 → 开始任务拆解
- Sprint 计划审核通过 → 开始开发

## Agent 行为规范

### 完成当前阶段后

```
✅ {阶段名称} 已完成！

📊 产出：
- ✅ {文档/工作项1}
- ✅ {文档/工作项2}

📝 下一步：
{下一阶段名称} - {简要说明}

❓ 要继续下一阶段吗？

[继续下一阶段] [暂停] [结束]
```

### 用户说"开始写代码"时

```
⚠️ 在开始写代码之前，需要先完成以下准备工作：

1. ✅ 产品设计（PRD）- {已完成/未完成}
2. ⬜ 技术设计 - {待完成}
3. ⬜ 任务拆解与排期 - {待完成}

❓ 请问：
1. 要从哪个阶段开始？
2. 还是跳过某些阶段？（不推荐）

[开始技术设计] [开始任务拆解] [直接开发（不推荐）]
```

## 常见场景处理

### 场景 1：用户直接说"帮我写代码"

```
⚠️ 在开始写代码之前，需要先完成以下准备工作：

1. ⬜ 产品设计（PRD）
2. ⬜ 技术设计
3. ⬜ 任务拆解与排期

❓ 请问：
1. 是否已有 PRD 和技术设计文档？
   A. 有，直接开始任务拆解
   B. 没有，从产品设计开始
   C. 没有，从技术设计开始

2. 还是跳过某些阶段？（不推荐）

请选择：___
```

### 场景 2：用户说"继续上次的工作"

```
📋 查找上次的工作记录...

上次完成的阶段：{阶段名称}
产出：{文档/工作项}

📝 下一步：{下一阶段名称}

❓ 要继续吗？

[继续] [查看上次产出] [重新开始]
```

### 场景 3：用户说"我要做 Sprint 计划"

```
📋 Sprint 计划需要基于已有的需求和技术设计。

请确认：
1. ✅/❌ 是否已有 PRD？
2. ✅/❌ 是否已有技术设计文档？
3. ✅/❌ 需求工作项是否已创建？

❓ 请问：
1. 要先完成缺失的阶段吗？
2. 还是基于现有内容直接做 Sprint 计划？

[先完成前置阶段] [直接做计划] [取消]
```

## 快速入口

如果用户明确知道要做什么，可以跳过部分引导：

| 用户意图 | 直接进入 |
|----------|----------|
| "帮我设计产品" | Phase 1: 产品设计 |
| "写技术设计" | Phase 2: 技术设计 |
| "做 Sprint 计划" | Phase 3: 任务拆解 |
| "开始开发" | Phase 4: 开发执行 |
| "做代码审查" | Phase 4: 代码审查 |
| "准备发布" | Phase 6: 发布管理 |

**但仍需检查前置条件是否满足。**
