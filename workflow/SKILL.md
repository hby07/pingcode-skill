# PingCode 工作流 Skill

本目录包含基于 PingCode 的交互式工作流模板。
所有工作流遵循「文档优先」原则和「产品管理 vs 项目管理」的职责划分。

---

## 核心原则

### 1. Agent 是协作者，不是执行者
- Agent 读取需求，制定计划
- Agent 执行任务，但关键决策需用户确认
- Agent 报告结果，等待用户反馈

### 2. 文档优先
- PRD 必须先写入 PingCode Wiki
- 技术设计文档必须先写入 PingCode Wiki
- 文档审核通过后，才能创建正式工作项

### 3. 产品管理 vs 项目管理
- **产品管理**：回答"做什么"和"为什么做"
- **项目管理**：回答"怎么做"、"什么时候做"和"谁来做"
- 需求变更必须经过产品和项目双方评估

### 4. 项目操作必须确认
- 涉及项目的新建、关联、切换时，必须先询问用户
- 用户可能有多个项目，操作错误项目会导致数据混乱
- 标准流程：先询问项目选择 → 确认后再执行操作

## 阶段衔接规则

### 规则 1：顺序执行
- Agent 必须按阶段顺序引导用户
- 不得跳过任何阶段

### 规则 2：阶段完成确认
- 每个阶段完成后，Agent 必须显示当前进度
- 等待用户确认后进入下一阶段

### 规则 3：前置条件检查
- 如果用户要求跳过阶段，Agent 必须检查前置条件
- 前置条件不满足时，拒绝跳过并解释原因

### 规则 4：进度可视化
每个阶段完成时，显示完整进度：

```
📊 当前进度：
- ✅ Phase 1: 需求收集与分析（产品管理）
- ✅ Phase 2: 方案设计（产品管理 + 技术团队）
- ✅ Phase 3: 迭代规划与任务拆解（项目管理）← 你在这里
- ⬜ Phase 4: 开发与测试（项目管理）
- ⬜ Phase 5: 发布与上线（项目管理 + 运维）
- ⬜ Phase 6: 上线验收与复盘（产品管理 + 项目管理）

📝 下一步：开发与测试
```

## 需求变更管理

### 变更流程
1. **产品经理**评估变更的必要性和影响
2. **产品经理**调整需求优先级（Product Backlog）
3. **项目经理**评估对迭代的影响
4. **项目经理**调整迭代计划和任务分配
5. **团队**执行变更

### 变更原则
- 变更必须经过评审
- 变更必须记录在案
- 变更必须通知所有相关方

## 产品管理 vs 项目管理 详细对比

| 维度 | 产品管理 | 项目管理 |
|------|---------|---------|
| **核心问题** | 做什么？为什么做？ | 怎么做？什么时候做？谁来做？ |
| **主要职责** | 需求管理、优先级排序、路线图 | 迭代规划、任务拆解、进度跟踪 |
| **核心工作项** | 需求、用户故事 | 迭代、任务、缺陷 |
| **核心视图** | Product Backlog、Roadmap | Sprint Board、Gantt Chart |
| **关键指标** | 需求完成率、用户满意度 | 迭代 velocity、任务完成率 |
| **时间维度** | 中长期（版本、季度） | 短期（迭代、周） |
