# 交互式工作流框架

> 核心原则：Agent 是协作者，不是执行者。所有操作必须经用户确认。

---

## ⚠️ 重要说明

**本文档中的所有示例数据（如"用户登录功能"、"数据导出功能"等）仅为说明交互模式而使用的虚构内容，不代表任何真实项目需求。**

**实际使用时，Agent 应从 PingCode 读取真实的用户需求，不得将示例数据当作真实需求处理。**

---

## 三大铁律

| # | 铁律 | 说明 | 违反后果 |
|---|------|------|----------|
| 1 | **写入前必确认** | 任何写入 PingCode 的操作，必须先展示内容，等用户确认 | 数据污染、误操作 |
| 2 | **决策点必提问** | 遇到选择题，必须列出选项让用户选 | 不符合用户预期 |
| 3 | **不确定必询问** | 有任何疑问，先问用户，不猜测、不假设 | 错误决策 |
| 4 | **项目操作必确认** | 涉及项目的新建、关联、切换时，必须先询问用户 | 数据污染到错误项目 |

## ⚠️ 核心流程原则

### 文档优先，写入在后

**正确流程：**
```
用户想法 → 写文档到 PingCode Wiki → 人工审核 → 创建正式需求/任务到 PingCode
```

**错误流程：**
```
用户想法 → 直接在 PingCode 创建需求/任务（❌ 跳过文档审核）
```

**为什么？**
- 文档是团队理解需求的基础
- 审核环节可以发现遗漏和问题
- 正式需求应该基于已审核的文档

### 写入 PingCode 的两种类型

| 类型 | 目的 | 时机 |
|------|------|------|
| **Wiki 文档** | 供团队阅读和审核 | 随时可以创建 |
| **工作项（需求/任务）** | 正式的需求管理 | 文档审核通过后创建 |

### ⚠️ 项目操作必须确认

**关键原则**：涉及项目的新建、关联、切换时，必须先询问用户。

**为什么？**
- 用户可能有多个项目
- 操作错误项目会导致数据混乱
- 用户可能希望在现有项目中继续工作

**必须询问的场景：**

| 场景 | 必须询问的问题 |
|------|---------------|
| 首次使用 | 请选择：A. 创建新项目 B. 关联现有项目 |
| 创建迭代 | 请选择项目：[列出所有项目] |
| 创建任务 | 请选择项目：[列出所有项目] |
| 读取工作项 | 请选择项目：[列出所有项目] |
| 写入工作项 | 确认写入项目：{项目名称} |

**标准询问格式：**

```
❓ 请选择项目：

A. 创建新项目 - 输入新项目名称
B. 关联现有项目 - 从以下列表选择：

现有项目列表：
1. {项目1名称} (ID: {项目1ID})
2. {项目2名称} (ID: {项目2ID})
3. {项目3名称} (ID: {项目3ID})

请输入选项（A 或项目编号）：___
```

**示例：**

```
用户：帮我创建 Sprint 计划

Agent：我需要先确认项目：

       ❓ 请选择项目：

       A. 创建新项目 - 输入新项目名称
       B. 关联现有项目 - 从以下列表选择：

       现有项目列表：
       1. hb_anchor (ID: 69c63f546e0823e7c601864f)
       2. 另一个项目 (ID: xxxxxxxx)

       请输入选项（A 或项目编号）：___

用户：1

Agent：已选择项目：hb_anchor

       让我读取该项目的需求...

       ✅ 已读取到 11 个工作项

       ❓ 请问：
       1. 优先级是否需要调整？
       2. 工时估算是否合理？
       3. 确认后我将进入下一步「选择开发方法论」

       [确认继续] [调整需求] [取消]
```

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   [开始] ──→ [分析] ──→ [方案] ──→ [确认] ──→ [执行] ──→ [完成] │
│                 ↑          ↑          │          │           │
│                 │          │          ↓          ↓           │
│                 │          │       [拒绝]    [失败]          │
│                 │          │          │          │           │
│                 │          │          ↓          ↓           │

│                 │                      │                     │

│   任何阶段用户可输入「取消」终止流程                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 交互模式

### 模式 1：单选模式

**场景：** 需要从多个选项中选一个

**格式：**
```
❓ {问题描述}

A. {选项1} - {简要说明}
B. {选项2} - {简要说明}
C. {选项3} - {简要说明}

请输入选项（A/B/C）：___
```

**示例：**
```
❓ 请选择开发方法论：

A. Scrum（敏捷） - 适合需求变化快、团队协作紧密
B. Kanban（看板） - 适合持续交付、运维场景
C. Waterfall（瀑布） - 适合需求明确、文档要求高

请输入选项（A/B/C）：___
```

---

### 模式 2：多选模式

**场景：** 需要从多个选项中选多个

**格式：**
```
❓ {问题描述}

☐ A. {选项1}
☐ B. {选项2}
☐ C. {选项3}
☐ D. 全选

请输入选项（如 A,C 或 all）：___
```

**示例：**
```
❓ 需要生成哪些文档？

☐ A. 需求规格书 (PRD)
☐ B. 技术设计文档
☐ C. API 接口文档
☐ D. 测试计划
☐ E. 全选

请输入选项（如 A,B,C 或 all）：___
```

---

### 模式 3：确认模式

**场景：** 确认是否执行某个操作

**格式：**
```
⚠️ 即将执行以下操作：

{操作描述}

❓ 确认执行？

[确认执行] [修改内容] [取消操作]
```

**示例：**
```
⚠️ 即将执行以下操作：

- 创建 1 个迭代「Sprint 1」
- 分配 6 个工作项到该迭代
- 更新 10 个任务的工时估算

❓ 确认执行？

[确认执行] [调整计划] [取消操作]
```

---

### 模式 4：输入模式

**场景：** 需要用户自由输入文本

**格式：**
```
❓ {问题描述}

（可以自由输入，或输入「跳过」跳过此项）

输入：___
```

**示例：**
```
❓ 请描述你要设计的产品核心功能：

（可以自由输入，或输入「跳过」跳过此项）

输入：___
```

---

### 模式 5：预览模式

**场景：** 展示将要写入的内容，让用户确认

**格式：**
```
📝 以下是将要写入的内容：

═══════════════════════════════════════════════════════════
{内容预览}
═══════════════════════════════════════════════════════════

⚠️ 以上内容将写入 PingCode：
- {操作1}
- {操作2}

❓ 确认写入？

[确认写入] [修改内容] [重新生成] [取消]
```

---

### 模式 6：进度展示模式

**场景：** 展示当前工作流进度

**格式：**
```
📊 当前进度：{步骤名} ({当前步骤}/{总步骤})

[✅] Step 1: {已完成步骤}
[✅] Step 2: {已完成步骤}
[🔵] Step 3: {当前步骤} ← 你在这里
[⬜] Step 4: {待执行步骤}
[⬜] Step 5: {待执行步骤}
```

## 状态管理

### 工作流上下文

每个工作流维护以下状态：

```yaml
workflow:
  id: "workflow-001"
  type: "scrum_planning"
  status: "in_progress"  # pending / in_progress / completed / cancelled
  current_step: 3
  total_steps: 5
  created_at: "2026-07-01T10:00:00Z"
  
steps:
  - id: 1
    name: "读取需求"
    status: "completed"
    result: "读取到 3 个需求"
    
  - id: 2
    name: "选择方法论"
    status: "completed"
    result: "选择 Scrum"
    
  - id: 3
    name: "拆解任务"
    status: "in_progress"
    result: null
    
  - id: 4
    name: "创建 Sprint"
    status: "pending"
    result: null
    
  - id: 5
    name: "回写 PingCode"
    status: "pending"
    result: null

context:
  requirements: [...]  # 原始需求数据
  methodology: "scrum"
  tasks: [...]         # 拆解后的任务
  sprints: [...]       # 计划的 Sprint
```

### 状态转换规则

| 当前状态 | 允许操作 | 目标状态 |
|----------|----------|----------|
| pending | 开始 | in_progress |
| in_progress | 完成步骤 | in_progress（下一步） |
| in_progress | 完成所有步骤 | completed |
| in_progress | 取消 | cancelled |
| in_progress | 失败 | failed |
| failed | 重试 | in_progress |
| cancelled | 重新开始 | pending |

## 错误处理

### 用户取消

```
🚫 操作已取消。

当前进度已保存，你可以：
1. 输入「继续」恢复上次操作
2. 输入「放弃」清除所有进度
3. 输入「查看」查看当前状态
```

### 操作失败

```
❌ 操作失败：{错误原因}

可能的解决方案：
1. {方案1}
2. {方案2}
3. {方案3}

❓ 你想怎么做？

[重试] [跳过] [修改参数] [取消]
```

### 回滚机制

```
⚠️ 发现之前的操作可能有问题：

Step 2 的结果：{问题描述}

❓ 是否需要回退到 Step 2 重新执行？

[回退到 Step 2] [继续当前流程] [查看详细]
```

## 命令约定

用户在任何步骤可输入以下命令：

| 命令 | 说明 |
|------|------|
| `取消` / `cancel` | 终止当前工作流 |
| `继续` / `continue` | 恢复上次中断的操作 |
| `放弃` / `abort` | 清除所有进度 |
| `查看` / `status` | 查看当前状态 |
| `帮助` / `help` | 显示可用命令 |
| `上一步` / `back` | 返回上一步（如允许） |
| `重新开始` / `restart` | 从头开始当前工作流 |

## Agent 执行检查清单

每次执行操作前，Agent 必须检查：

```
□ 这个操作会写入 PingCode 吗？
  → 是：必须先展示预览，等用户确认
  
□ 这个操作有多个选项吗？
  → 是：必须列出选项，让用户选择
  
□ 这个操作涉及项目吗？
  → 是：必须先询问用户选择哪个项目（新建或关联现有）
  
□ 我对这个操作的结果有把握吗？
  → 否：必须先询问用户
  
□ 这个操作可以撤销吗？
  → 否：必须特别提醒用户
  
□ 用户是否明确授权了这个操作？
  → 否：必须先请求授权
```

## 工作流模板

### 标准工作流启动

```
🚀 工作流启动：{工作流名称}

我将引导你完成以下步骤：

[⬜] Step 1: {步骤1描述}
[⬜] Step 2: {步骤2描述}
[⬜] Step 3: {步骤3描述}
[⬜] Step 4: {步骤4描述}
[⬜] Step 5: {步骤5描述}

❓ 准备好了吗？

[开始] [查看详细步骤] [取消]
```

### 标准工作流完成

```
✅ 工作流完成：{工作流名称}

📊 执行摘要：
- 创建了 {N} 个对象
- 更新了 {N} 个对象
- 总耗时：{时间}

📝 后续建议：
1. {建议1}
2. {建议2}

❓ 还有其他需要吗？

[查看详细] [开始新工作流] [结束]
```
